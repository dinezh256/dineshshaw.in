## आपका डेटा बाहर जाने से पहले टैब को बंद न होने दें

यहाँ एक दिलचस्प स्थिति है: उपयोगकर्ता एक लंबा फॉर्म भरता है, उसका ध्यान भटक जाता है, और वह टैब बंद कर देता है। आपके बैकएंड को कभी कुछ पता ही नहीं चलता। आपके ऐप के आधार पर, यह मामूली परेशानी से लेकर एक गंभीर डेटा अखंडता (data integrity) समस्या तक कुछ भी हो सकता है।

मैंने इस समस्या का सामना तब किया जब मैं एक ऐसे फीचर पर काम कर रहा था जो किसी पृष्ठ पर बिताए गए समय को ट्रैक करता था, इस तरह की चीज़ जिसे *ठीक उसी समय* फ़ायर करने की आवश्यकता होती है जब उपयोगकर्ता पृष्ठ छोड़ता है, न कि किसी टाइमर पर। सामान्य तरीके इसमें काम नहीं आए।

### भोला-भाला तरीका और यह क्यों विफल रहता है

मेरी पहली सोच थी कि बस एक `beforeunload` लिसनर के अंदर `fetch` का उपयोग कर लूँ:

```js
window.addEventListener('beforeunload', () => {
  fetch('/api/track', { method: 'POST', body: JSON.stringify(data) });
});
```

यह मज़बूती से काम नहीं करता है। ब्राउज़र पेज अनलोड होने के दौरान async संचालन (operations) की प्रतीक्षा नहीं करता है, इसलिए अनुरोध (request) पूरा होने से पहले ही रद्द हो जाता है। आप कभी-कभी भाग्यशाली हो सकते हैं, लेकिन आप इस पर भरोसा नहीं कर सकते।

### Beacon API का प्रवेश

`navigator.sendBeacon()` API ठीक इसी उपयोग के मामले (use case) के लिए मौजूद है। यह पृष्ठभूमि में एक छोटा HTTP POST भेजता है, और महत्वपूर्ण रूप से, ब्राउज़र *गारंटी* देता है कि पेज के बंद होने के दौरान भी इसे भेजा जाएगा। कोई प्रतीक्षा नहीं, कोई रुकावट (blocking) नहीं।

```js
navigator.sendBeacon('/api/track', JSON.stringify(data));
```

बस इतना ही। बाकी सब ब्राउज़र संभाल लेता है। यह नॉन-ब्लॉकिंग है, पेज को बंद होने में देरी नहीं करता है, और सभी आधुनिक ब्राउज़रों में समर्थित है।

### इसे React में जोड़ना

यहाँ बताया गया है कि मैंने इसे `useEffect` का उपयोग करके एक component के अंदर कैसे सेट अप किया:

```jsx
import { useEffect } from 'react';

const useBeaconOnLeave = (data) => {
  useEffect(() => {
    const handleBeforeUnload = () => {
      navigator.sendBeacon('/api/track', JSON.stringify(data));
    };

    window.addEventListener('beforeunload', handleBeforeUnload);

    return () => {
      window.removeEventListener('beforeunload', handleBeforeUnload);
    };
  }, [data]);
};
```

इसे एक कस्टम हुक में लपेटें, वह डेटा पास करें जिसे आप ट्रैक करना चाहते हैं, और आपका काम हो गया। return में क्लीनअप महत्वपूर्ण है। यदि component किसी अन्य कारण से अनमाउंट होता है (उदाहरण के लिए, SPA के भीतर नेविगेशन), तो आप नहीं चाहेंगे कि एक पुराना लिसनर लटका रहे।

### कुछ बातें ध्यान में रखने योग्य

**पेलोड (payload) की एक आकार सीमा है।** `sendBeacon` बड़े डेटा के लिए नहीं है। इसे छोटे JSON ऑब्जेक्ट्स तक सीमित रखें: सेशन डेटा, एनालिटिक्स इवेंट्स, इस तरह की चीज़ें। इसके माध्यम से अपने ऐप की पूरी स्थिति (state) डंप करने की कोशिश न करें।

**यह केवल सच्चे पेज अनलोड पर फ़ायर होता है।** एक React ऐप के अंदर रूट (Route) परिवर्तन `beforeunload` को ट्रिगर नहीं करते हैं, क्योंकि वे ब्राउज़र द्वारा नहीं, React Router द्वारा नियंत्रित होते हैं। यदि आपको इन-ऐप नेविगेशन को ट्रैक करने की आवश्यकता है, तो आप इसके बजाय अपने राउटर के `history` में हुक करना चाहेंगे।

**सर्वर का अभी भी सुरक्षित होना आवश्यक है।** केवल इसलिए कि क्लाइंट डेटा भेज रहा है, इसका मतलब यह नहीं है कि आप उस पर आँख बंद करके भरोसा कर लें। सर्वर साइड पर उसी तरह से सत्यापित (validate), प्रमाणित (authenticate) और सैनिटाइज़ करें जैसे आप किसी अन्य अनुरोध के साथ करेंगे।

### आपको वास्तव में इसका उपयोग कब करना चाहिए?

ईमानदारी से कहूं तो, अक्सर नहीं। अधिकांश ऐप्स को इसकी आवश्यकता नहीं होती है। लेकिन यदि आप इनमें से कोई भी काम कर रहे हैं, तो इसके बारे में जानना उचित है:

- एनालिटिक्स या सहभागिता (engagement) ट्रैकिंग (पेज पर समय, स्क्रॉल की गहराई)
- ड्राफ्ट स्थिति को ऑटो-सेव करना
- सेशन समाप्त होने के इवेंट्स को लॉग करना
- कोई भी "उनके छोड़ने से पहले की अंतिम क्रिया" (last action before they leave) प्रकार की सुविधा

यह एक संकीर्ण उपयोग का मामला है, लेकिन जब आपको इसकी आवश्यकता होती है, तो आपको वास्तव में इसकी आवश्यकता होती है। और अब आप काम के लिए सही उपकरण जानते हैं।
