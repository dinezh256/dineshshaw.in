## कीबोर्ड उपयोगकर्ता मौजूद हैं। यहाँ बताया गया है कि मैंने पॉपअप में उन्हें नज़रअंदाज़ करना कैसे बंद किया।

एक्सेसिबिलिटी (Accessibility) को अक्सर बाद में सोचा जाता है, या इससे भी बदतर, एक चेकबॉक्स की तरह माना जाता है। मैं भी इसका दोषी था। जब तक कि किसी ने मेरे बनाए एक फीचर के माध्यम से स्क्रीन रीडर नहीं चलाया और कुछ भी समझ में नहीं आया। पॉपअप खुल गए, फ़ोकस उस बटन पर बना रहा जिसने उन्हें ट्रिगर किया था, और स्क्रीन रीडर बस... पॉपअप के पीछे जो कुछ भी था उसे पढ़ता रहा।

सुधार जटिल नहीं है, लेकिन अगर आप इसके बारे में नहीं सोच रहे हैं तो इसे छोड़ देना आसान है।

### वास्तविक समस्या क्या है?

जब कोई उपयोगकर्ता किसी बटन पर क्लिक करता है और एक पॉपअप खुलता है, तो उनका कीबोर्ड फ़ोकस बटन पर ही रहता है। माउस उपयोगकर्ताओं के लिए, यह ठीक है। वे पॉपअप देख सकते हैं और उसमें क्लिक कर सकते हैं। लेकिन केवल-कीबोर्ड उपयोगकर्ताओं या स्क्रीन रीडर उपयोगकर्ताओं के लिए, पॉपअप का कोई अस्तित्व नहीं है। वे Tab दबाते हैं और पॉपअप के अंदर जाने के बजाय पॉपअप के *पीछे* जो कुछ भी है, उसके माध्यम से नेविगेट करते रहते हैं।

सुधार: जब पॉपअप खुलता है, तो प्रोग्रामेटिक रूप से फ़ोकस को इसके *अंदर* ले जाएं।

### सेटअप

मैं Semantic UI के `Popup` component का उपयोग कर रहा था, लेकिन यह तरीका किसी भी पॉपअप लाइब्रेरी के साथ काम करता है। मुख्य बातें ये हैं:

1. पॉपअप सामग्री पर एक `ref` ताकि हम उस पर `.focus()` कॉल कर सकें
2. एक `useEffect` जो पॉपअप के खुलने पर नज़र रखता है और फ़ोकस को ट्रिगर करता है

```jsx
import React, { useState, useEffect, useRef } from "react";
import { Button, Popup } from "semantic-ui-react";

const MyComponent = () => {
  const [openPopup, setOpenPopup] = useState(false);
  const popupRef = useRef(null);

  useEffect(() => {
    if (openPopup && popupRef.current) {
      // एक छोटे टाइमआउट की आवश्यकता है: पॉपअप को DOM में रेंडर होने के लिए एक टिक की आवश्यकता होती है
      setTimeout(() => popupRef.current?.focus(), 0);
    }
  }, [openPopup]);

  return (
    <Popup
      trigger={<Button onClick={() => setOpenPopup(true)}>Open</Button>}
      content={
        <Popup.Content tabIndex="0" ref={popupRef}>
          Popup Content
        </Popup.Content>
      }
      on="click"
      open={openPopup}
      onClose={() => setOpenPopup(false)}
    />
  );
};
```

दो बातें ध्यान देने योग्य हैं:

पॉपअप सामग्री पर **`tabIndex="0"`** आवश्यक है। डिफ़ॉल्ट रूप से, `div`s और अधिकांश गैर-संवादात्मक (non-interactive) तत्व फ़ोकस प्राप्त नहीं कर सकते हैं। `tabIndex="0"` सेट करने से यह टैब क्रम (tab order) में रखे बिना फ़ोकस करने योग्य बन जाता है।

**`setTimeout`** एक तरह का जुगाड़ (hack) है, लेकिन एक आवश्यक जुगाड़ है। जब पॉपअप खुलता है, तो यह ज़रूरी नहीं है कि `useEffect` के चलने तक यह DOM में रेंडर हो गया हो। देरी की एक टिक React को अपडेट पूरा करने का समय देती है।

### पॉपअप बंद करने के बारे में क्या?

समान रूप से महत्वपूर्ण: जब पॉपअप बंद हो जाता है, तो फ़ोकस उस तत्व पर वापस आ जाना चाहिए जिसने इसे ट्रिगर किया था। अन्यथा उपयोगकर्ता का फ़ोकस शून्यता (void) में गायब हो जाता है।

```jsx
const triggerRef = useRef(null);

const handleClose = () => {
  setOpenPopup(false);
  triggerRef.current?.focus();
};

// ट्रिगर बटन पर:
<Button ref={triggerRef} onClick={() => setOpenPopup(true)}>Open</Button>
```

यह पूरा लूप है: पॉपअप खुलने पर फ़ोकस अंदर चला जाता है, और बंद होने पर ट्रिगर पर वापस आ जाता है। वापसी के बिना, कीबोर्ड उपयोगकर्ता पॉपअप को हटाने के बाद फँसे रह जाते हैं।

### क्या यह सिर्फ एक अच्छा अभ्यास है या वास्तव में आवश्यक है?

तकनीकी रूप से, WCAG 2.1 अनुपालन (विशेष रूप से [Success Criterion 2.4.3](https://www.w3.org/WAI/WCAG21/Understanding/focus-order.html)) के लिए, मोडल डायलॉग और पॉपअप में फ़ोकस प्रबंधन अपेक्षित है। यदि आपका उत्पाद किसी भी उद्यम (enterprise) या सरकारी क्लाइंट को सेवा प्रदान करता है, या यदि आप अपने उपयोगकर्ताओं के एक बड़े हिस्से को बाहर नहीं करना चाहते हैं, तो यह मायने रखता है।

और सच कहूँ तो, यह बहुत अधिक कोड नहीं है। इसे छोड़ने का कोई अच्छा कारण नहीं है।
