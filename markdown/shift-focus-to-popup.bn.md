## কীবোর্ড ব্যবহারকারীদেরও অস্তিত্ব আছে। আমি কীভাবে পপআপে তাদের অবহেলা করা বন্ধ করলাম।

অ্যাক্সেসিবিলিটিকে (Accessibility) প্রায়শই একটি পরের ভাবনা হিসাবে বিবেচনা করা হয়, বা এর চেয়েও খারাপ, একটি চেকবক্সের মতো আচরণ করা হয়। আমিও এই ভুলটি করেছি। যতক্ষণ না কেউ আমার তৈরি করা একটি ফিচারের মাধ্যমে স্ক্রিন রিডার ব্যবহার করে এবং কিছুই বুঝতে পারে না। পপআপ খুলেছিল, ফোকাস সেই বোতামটিতেই থেকে গিয়েছিল যা তাদের ট্রিগার করেছিল এবং স্ক্রিন রিডারটি শুধু... পপআপের পিছনে যা ছিল তা পড়তে থাকে।

সমাধানটি জটিল নয়, তবে আপনি যদি এর কথা না ভাবেন তবে এটি এড়িয়ে যাওয়া সহজ।

### আসল সমস্যাটা কী?

যখন কোনো ব্যবহারকারী একটি বোতামে ক্লিক করে এবং একটি পপআপ খোলে, তখন তাদের কীবোর্ডের ফোকাস বোতামটিতেই থাকে। মাউস ব্যবহারকারীদের জন্য, এটি ঠিক আছে। তারা পপআপটি দেখতে পারে এবং তার মধ্যে ক্লিক করতে পারে। কিন্তু শুধুমাত্র কীবোর্ড ব্যবহারকারী বা স্ক্রিন রিডার ব্যবহারকারীদের জন্য, পপআপটি যেন অস্তিত্বহীন। তারা Tab চাপ দেয় এবং পপআপের ভিতরে যাওয়ার পরিবর্তে পপআপের *পিছনে* যা আছে তাতেই নেভিগেট করতে থাকে।

সমাধান: পপআপ খোলার সময়, প্রোগ্রামাটিকভাবে ফোকাসটিকে এর *ভিতরে* সরিয়ে নিন।

### সেটআপ

আমি Semantic UI-এর `Popup` component ব্যবহার করছিলাম, তবে এই পদ্ধতি যেকোনো পপআপ লাইব্রেরির সাথে কাজ করে। মূল বিষয়গুলো হলো:

1. পপআপ সামগ্রীতে একটি `ref` যাতে আমরা সেটিতে `.focus()` কল করতে পারি
2. একটি `useEffect` যা পপআপ খোলার সময় নজর রাখে এবং ফোকাস ফায়ার করে

```jsx
import React, { useState, useEffect, useRef } from "react";
import { Button, Popup } from "semantic-ui-react";

const MyComponent = () => {
  const [openPopup, setOpenPopup] = useState(false);
  const popupRef = useRef(null);

  useEffect(() => {
    if (openPopup && popupRef.current) {
      // একটি ছোট টাইমআউটের প্রয়োজন: পপআপটিকে DOM-এ রেন্ডার হতে এক টিক সময় লাগে
      setTimeout(() => popupRef.current?.focus(), 0);
    }
  }, [openPopup]);

  return (
    <Popup
      trigger={<Button onClick={() => setOpenPopup(true)}>Open</Button>}
      content={
        <Popup.Content tabIndex="0" ref={popupRef}>
          Popup Content
        </Popup.Content>
      }
      on="click"
      open={openPopup}
      onClose={() => setOpenPopup(false)}
    />
  );
};
```

দুটি বিষয় উল্লেখ করার মতো:

পপআপ কন্টেন্টে **`tabIndex="0"`** দেওয়া আবশ্যক। ডিফল্টভাবে, `div` এবং বেশিরভাগ নন-ইন্টারেক্টিভ (non-interactive) উপাদান ফোকাস পেতে পারে না। `tabIndex="0"` সেট করলে এটি ট্যাব অর্ডারে (tab order) না রেখেই ফোকাসযোগ্য হয়ে ওঠে।

**`setTimeout`** হলো এক ধরণের হ্যাক (hack), তবে এটি একটি প্রয়োজনীয় হ্যাক। যখন পপআপ খোলে, তখন `useEffect` চালনার সময় এটি DOM-এ রেন্ডার না-ও হতে পারে। এক টিক দেরি React-কে আপডেটটি সম্পন্ন করার সময় দেয়।

### পপআপ বন্ধ করার বিষয়ে কী?

একইভাবে গুরুত্বপূর্ণ: যখন পপআপটি বন্ধ হয়ে যায়, ফোকাসটি সেই উপাদানটিতে ফিরে আসা উচিত যা এটি ট্রিগার করেছিল। অন্যথায় ব্যবহারকারীর ফোকাস শূন্যতায় (void) হারিয়ে যায়।

```jsx
const triggerRef = useRef(null);

const handleClose = () => {
  setOpenPopup(false);
  triggerRef.current?.focus();
};

// ট্রিগার বাটনে:
<Button ref={triggerRef} onClick={() => setOpenPopup(true)}>Open</Button>
```

এটি হলো পুরো লুপ: পপআপ খোলার সময় ফোকাস ভিতরে যায় এবং বন্ধ হওয়ার সময় ট্রিগারে ফিরে আসে। এটি না করলে, কীবোর্ড ব্যবহারকারীরা পপআপটি সরানোর পর আটকে থাকেন।

### এটি কি শুধুই একটি ভালো অভ্যাস নাকি আসলেই প্রয়োজনীয়?

প্রযুক্তিগতভাবে, WCAG 2.1 কমপ্লায়েন্সের (বিশেষ করে [Success Criterion 2.4.3](https://www.w3.org/WAI/WCAG21/Understanding/focus-order.html)) জন্য মোডাল ডায়ালগ এবং পপআপে ফোকাস ম্যানেজমেন্ট প্রত্যাশিত। যদি আপনার পণ্য কোনো এন্টারপ্রাইজ (enterprise) বা সরকারি ক্লায়েন্টকে পরিষেবা প্রদান করে, বা আপনি যদি আপনার ব্যবহারকারীদের একটি বড় অংশকে বাদ দিতে না চান, তবে এটি গুরুত্বপূর্ণ।

আর সত্যি বলতে, এর জন্য খুব বেশি কোড লাগে না। এটি এড়িয়ে যাওয়ার কোনো যুক্তিসঙ্গत কারণ নেই।
